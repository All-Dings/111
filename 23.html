Topics: TeX, Dings-Tool, History

For my last Blog-Entry I introduced into the Dings-Project the Markup-Language TeX, which especially allows to render Mathematical-Symbols in a quite beatiful Way.

TeX has been invented between the 1970th and 80th by the Computer-Legend Donald_Knuth and then it was used in the early 80th by Leslie_Lamport to implement his Lamport-TeX or shortly LaTeX. The latter is the most widespread today and also I have used it extensively for my Studien-Arbeit in 1995/96.

Here is a short TeX Code-Fragment for an Equation, …

$$ H = \sqrt[12]{2} \approx 1,0595 $$

… which is then rendered as:

H=212≈1,0595 H = \sqrt[12]{2} \approx 1,0595

A single “$” starts and ends the Math-Mode for TeX and a double “$$” is used for centered unlabled Math-Equations. Fortunately the Pandoc-Tool allows us to use these Formulas also in Markdown-Files.

From the different Pandoc-Options for translating TeX into Html I use the --mathml Option that, as the Name indicates, produces MathMl as Output.

The upper Example is translated then into the following Mathematical-Markup-Language:

<math display="block" xmlns="http://www.w3.org/1998/Math/MathML">
 <semantics>
  <mrow>
   <mi>H</mi><mo>=</mo>
   <mroot>
    <mn>2</mn>
    <mn>12</mn>
   </mroot>
   <mo>=</mo>
   <mn>1</mn><mo>,</mo><mn>0595</mn>
  </mrow>
  <annotation encoding="application/x-tex"> H = \sqrt[12]{2} = 1,0595 </annotation>
  </semantics>
</math>

As we already saw, the rendered Result already looks quite nice, but unfurtunately there is no Support for Labeling and Referencing the Equations. Therefore I extend the Dings-Tool once more and introduce a new Syntax for appending a Label and an Anchor:

$$ Latex-Expression $${#LABEL:ANCHOR}

For a nice Formatting I use a Html-Table with three Columns, putting the Equation in the middle and the Anchor to the right. I also add new Css-Classes, like “TeX-Table” or “TeX-Tr” to the Dings-Css-File.

The following Example …

$$ H = \sqrt[12]{2} \approx 1,0595 $${#0:10480}

… is then translated into this Html-Code, …

<table class="TeX-Table">
  <tr class="TeX-Tr">
   <td class="TeX-Td-Left">
     <a id="10480"/>
   </td>
   <td class="TeX-Td-Equation">
     <math display="block" xmlns="http://www.w3.org/1998/Math/MathML"><semantics><mrow><mi>H</mi><mo>=</mo><mroot><mn>2</mn><mn>12</mn></mroot><mo>≈</mo><mn>1</mn><mo>,</mo><mn>0595</mn></mrow><annotation encoding="application/x-tex"> H = \sqrt[12]{2} \approx 1,0595 </annotation></semantics></math>
   <td class="TeX-Td-Right">(0)</td>
  </tr>
</table>

… that is finally rendered as:

H=212≈1,0595 H = \sqrt[12]{2} \approx 1,0595 (0)

Day-101: Heimat-Programm & Pythagorean-Comma again

Blog-Chain-Time 2023.07.04-23:00 in Tübingen

Topics: Music, Mathematics

Because after on our Journey to Freiburg, I was not 100% sure, if my Thoughts about the Pythagorean-Comma were correct, I want to try a Mathematics-Proof now.

According to Wikipedia [0] the Pythagorean-Comma is 23,46 Cent and in the Equal-Temperament one Cent is defined as one hundredth of a Half-Tone.

Since in this System one Half-Tone is exactly one twelfth of an Octave [1], we have to apply to be Base-Frequency twelve Times the Multiplication of the Frequency-Ratio of H in order to reach the Octave O:

Base-Frequency*H*H*H*H*H*H*H*H*H*H*H*H=O\operatorname{Base-Frequency} * H * H * H * H * H * H * H * H * H * H * H * H = O

Because we choose BaseBase-Frequency=1Frequency=1 and we already know that the Frequency-Ratio of the Octave is 2:1 or simply 2, we can write this as:

H*H*H*H*H*H*H*H*H*H*H*H=2H * H * H * H * H * H * H * H * H * H * H * H = 2

Using the Exponentiation we can formulate this also much shorter as:

H12=2H^12 = 2

In order to calculate H, we have to apply Exponentiation with 112\frac{1}{12} on both Sides:

(H12)112=2112(H^{12})^{\frac{1}{12}} = 2^{\frac{1}{12}}

After shortening the Fraction 12*112=112*\frac{1}{12} = 1 and replacing 21122^{\frac{1}{12}} with the more common Root-Writing 212\sqrt[12]{2}, we get the following Frequency-Ratio for our Half-Tone:

H=212≈1,0595H = \sqrt[12]{2} \approx 1,0595

As mentioned at the Beginning, according to Wikipedia the Pythagorean-Comma is 23,46 Cent and in the Equal-Temperament one Cent is defined as one hundredth of a Half-Tone, so one could naively assume that we now can calculate Comma-Expected as:

Comma-Expected=23,46Cent*1100\operatorname{Comma-Expected} = 23,46\, Cent * \frac{1}{100}

Unfortunately it is not such simple, because in our Case we don’t have a linear System. Therefore the Calculation of the Frequence-Ratio for the Half-Tone was the wrong way, sorry by bad. Instead the Frequency-Ratio FR of CT Cents can be calculated [2] with:

FR=2CT1200CentFR = 2^{\frac{CT}{1200\, Cent}}

The Reason for the 1200 Cent is the Octave again, that is exactly 12 Have-Tones, or 12*100Cent=1200Cent12 * 100\, Cent = 1200\, Cent, so 21200Cent1200Cent=21=22^{\frac{1200\, Cent}{1200\, Cent}} = 2^1 = 2, which is - as we already know - the Frequency-Ratio 2:1 of our Ocatave.

So now let’s calculate the Frequency-Ratio for the Pythagorean-Comma with 23,46 Cent:

Comma-Expected=223,46Cent1200Cent≈1,0136\operatorname{Comma-Expected} = 2^{\frac{23,46\, Cent}{1200\, Cent}} \approx 1,0136 (1)

If we look at our Results for the Target-Frequencies of 12 Fiths (1,5121,5^{12}) and 7 Octaves (272^7), our self-calculated Pythagorean-Comma can be determinded by:

Comma-Calculated=1,51227=129,75128,00≈1,0136\operatorname{Comma-Calculated} = \frac{1,5^{12}}{2^{7}} = \frac{129,75}{128,00} \approx 1,0136 (2)

Since (1) and (2) result in the same Value, my Calculation seems to be correct and even more Digits behind the Comma look also quite promising:

But I still don’t have the Feeling to really have understood this Music-System completely, so probably it takes a little bit more Time until I can say “q. e. d.” with enough Confidence.

Topics: Hash-Functions

The last Days I have dealt with the Topic Hash-Functions: In Informatics you find them everywhere, for Example Git uses SHA-1 or Bitcoin utilizes SHA-256, which is one Member of the SHA-2-Family.

The Algorithms mentioned have been developed by the NSA and standardized by the NIST. Both Institutions are located in the USA.

Hash-Functions are used to reduce big Numbers to small Ones. A simple Secimen for an Hash-Algorithm is the Digit-Sum, where for Example the Number 4711 is mapped to the Number 13.

A Problem of Hash-Sums are the so called Hash-Collisions: Since the Input-Set is bigger than the Output-Set it is unavoidable that there are two Input-Numbers that map to the same Output-Number.

Here an Example for our Digit-Sum, where the Numbers 4711 and 1003 lead both to the Number 13:

Digit-Sum(4711) = 13 (4 + 7 + 1 + 1)

## Quick-Sort for List of Dictionary
def Quicksort_List_of_Dictionary(List, Key):
        Element_Count = len(List) 
        if Element_Count <= 1:
                return List
        Pivot = List[Element_Count // 2][Key]
        Left = [Element for Element in List if Element[Key] < Pivot]
        Middle = [Element for Element in List if Element[Key] == Pivot]
        Right = [Element for Element in List if Element[Key] > Pivot]
        return Quicksort_List_of_Dictionary(Left, Key) + Middle + Quicksort_List_of_Dictionary(Right, Key)

So, after thinking a little bit about Information-Sharing Theory, let’s have a Look on the History of Internet:

The British Man Tim_Berners-Lee invented HTML that provides Hyper-Links for Referencing Information in the World-Wide-Web.

A Link can be specified as follows:

<a href="https://holzheu.de/">Web-Site from Michael_Holzheu</a>

Like with Markdown, there seems to be no native Mechanism [3211] in HTML to include other HTML-Pages, but interestingly including Pictures is possible:

<img src="0.jpg" alt="Michael_Holzheu"/>

The US Man Ted_Nelson suggested an Alterntive to the Ideas of Tim_Berners-Lee and implemented the Xanadu-System, which is able to include Information in other Documents using so-called Transclusion, where the Data is copied only at Rendering-Time.

The Concept of Transclusion is a very wide-spread Thing that is called Mapping in Virtualization, e.g. the following three Examples immediately come into my Mind:

  • Mappings with the MMap-Unix-System-Call
  • References in Spread-Sheets
  • Include-Files in Programming-Languages

Here a small Diagram that illustrates the Concept of Virtual-Mapping at the Example of two different Virtual-Documents that are constructed out of three “real” Objects:

|       Virtual-1          Real           Virtual-2       |
|                                                         |

| |                   |             |                   | |
| |     Object-2      |  Object-2   |     Object-2      | |
| |                   |             |                   | |

| +-------------------+             +-------------------+ |
|                                                         |

Advantages of Virtual-Mappings:

  • Resource-Consumption: The References require less Ressources than the real Objects
  • Consistency: When a real Object is changed, all virtual Objects automatically get the Update

Disadvantages of Virtual-Mappings:

  • External-References: When mapping a real Object also all of its External-References have to be somewhere mapped in the Virtual-Document.
  • Reference-Relocation: If References have a Local-Name they have to be renamed in the Virtual-Document, this Process is also Sometimes called Rellocation.
  • Resource-Consumption: The Resolution of References in the Rendering-Process also takes some Time.
  • Consistency: When one Physical-Object is updated and the Meaning changes, the overall Consisteny in the Virtual-Document still have to be ensured somehow.

function Update_Sub_Modules()
{
        local Tag=$1
        local Old_Tag;

        for Git_Sub_Module in $(git submodule | awk '{print $2}')
        do
                >&2 echo "Updating: $Git_Sub_Module"

                Old_Tag=$(git tag -l | grep $Tag)
                cd $Git_Sub_Module
                git pull origin Master
                git checkout Master
                git pull --rebase

                if [ "$Old_Tag" != "" ]; then
                        git tag -d $Tag
                fi
                git push origin :refs/tags/$Tag
                git tag $Tag
                git push --tags;
                cd ..
        done
}

Currently we have three different Types of Information-Containers:

  • Repositories
  • Number-Files
  • Documents

Repositories contain Number-Files, Number-Files contain Documents, Documents contain e.g. Blog-Entries [1911]. We have the following Addressing for every Piece of Information that we can use for References from other Information:

  • Repository-Number
  • Number-File-Number
  • Html-Tag-Number

The first two Numbers currently are unique in the Scope of this Project, but the Html-Tag-Number is only unequivocal within the Document.

What makes the Things even more complicated is the Fact that Information Containers can change over Time. So when we e.g. reference a Blog-Entry from another Blog-Entry, after some Days the Content of the first one could have canged, which then makes the Reference from the second one somehow invalid.

There are several Solutions to that Problem:

  1. Use a Time-Stamp in the Reference
  2. Use a Hash-Value for the Information in the Reference
  3. Don’t use References and instead copy the Information
  4. Try to keep the whole Project consistent at one Point in Time

My current Approach to solve the Problem of Consistency is Solution 4. Therefore I try make one Day as consistent as possible. With Git it is still possible to checkout and render old Days. So if you are interested in the Truth of the Past you still can find it.
